# Auth e variáveis (cluster)

Substitua os placeholders: `YOUR_GCP_PROJECT_ID`, região e nome da service account conforme o seu ambiente.

## Opção A — sua conta (ADC com usuário)

```bash
gcloud auth application-default login
```

## Impersonação (sem JSON da service account)

Precisa de `roles/iam.serviceAccountTokenCreator` na service account que será impersonada.

```bash
gcloud auth application-default login \
  --impersonate-service-account=terraform@YOUR_GCP_PROJECT_ID.iam.gserviceaccount.com
```

## Opção B — service account do Terraform (chave JSON)

Use uma conta no formato `terraform@YOUR_GCP_PROJECT_ID.iam.gserviceaccount.com` (ou outro nome que você criou).

### 1. Chave JSON (uma vez)

```bash
gcloud iam service-accounts keys create ~/secrets/terraform-sa-key.json \
  --iam-account=terraform@YOUR_GCP_PROJECT_ID.iam.gserviceaccount.com \
  --project=YOUR_GCP_PROJECT_ID
```

Guarde o arquivo fora do repositório e **não** faça commit.

### 2. Apontar o provider Google do Terraform

```bash
export GOOGLE_APPLICATION_CREDENTIALS="$HOME/secrets/terraform-sa-key.json"
```

(Opcional: `export GOOGLE_CLOUD_PROJECT=YOUR_GCP_PROJECT_ID` — o `project` do `provider` também vem de `var.gcp_project_id`.)

### 3. IAM — papéis no projeto para essa service account

```bash
PROJECT_ID=YOUR_GCP_PROJECT_ID
SA=terraform@${PROJECT_ID}.iam.gserviceaccount.com

gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:${SA}" \
  --role="roles/compute.admin"

gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:${SA}" \
  --role="roles/container.admin"
```

- **`compute.admin`**: VPC, subnets, firewall, Cloud Router/NAT, etc.
- **`container.admin`**: GKE no Terraform (quando existir no módulo).

Se os nós usarem **outra** service account, costuma ser necessário `roles/iam.serviceAccountUser` nessa conta.

### 4. Variáveis do Terraform

`gcp_project_id` tem de ser o **ID do projeto em texto** (ex. `my-proj`), não o número — o GKE Workload Identity falha com número.

```bash
export TF_VAR_gcp_project_id="YOUR_GCP_PROJECT_ID"
export TF_VAR_gcp_region="YOUR_GCP_REGION"
```

Ou defina em `terraform.tfvars` local (sem commitar segredos).

### 5. Rodar

```bash
cd cluster
terraform plan
terraform apply
```

### 6. Conectar ao cluster com `kubectl`

Precisa de [`gcloud`](https://cloud.google.com/sdk/docs/install), do **plugin de autenticação GKE** e permissão **Kubernetes Engine Cluster Viewer** (ou superior) no projeto.

**Plugin (obrigatório em clusters recentes):** sem isto o `kubectl` mostra *gke-gcloud-auth-plugin was not found*. [Documentação](https://cloud.google.com/kubernetes-engine/docs/how-to/cluster-access-for-kubectl#install_plugin).

- **Ubuntu/Debian (gcloud instalado com `apt`)** — o gestor de componentes do `gcloud` costuma estar desligado; use o pacote:

```bash
sudo apt-get update && sudo apt-get install -y google-cloud-cli-gke-gcloud-auth-plugin
```

(Pacote antigo: `google-cloud-sdk-gke-gcloud-auth-plugin`.)

- **Instalação oficial do SDK (component manager ativo):**

```bash
gcloud components install gke-gcloud-auth-plugin
```

Autenticação de utilizador:

```bash
gcloud auth login
```

No diretório `cluster/` (com `terraform apply` já feito e state atual):

```bash
gcloud container clusters get-credentials "$(terraform output -raw gke_cluster_name)" \
  --region "$(terraform output -raw gcp_region)" \
  --project "$(terraform output -raw gcp_project_id)"
```

Isto atualiza o kubeconfig (contexto por defeito passa a ser o deste cluster). Testar:

```bash
kubectl cluster-info
kubectl get nodes
```

Equivalente manual (substitua pelos valores do `terraform output` ou do `terraform.tfvars`):

```bash
gcloud container clusters get-credentials NOME_DO_CLUSTER \
  --region REGIAO \
  --project SEU_PROJECT_ID
```
